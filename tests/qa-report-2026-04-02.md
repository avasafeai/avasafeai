## VISUAL ISSUES

1. **Apply form faded on load (CRITICAL UX):** /apply/form step 1 — the entire form (heading "Your name", field labels, input borders, all text) renders in near-invisible off-white/light grey on initial page load. This is Framer Motion's initial animation state (`opacity: 0` or close to it) being captured before the enter animation fires. In normal use the animation probably completes in ~300ms, but it creates a jarring blank-looking screen if the animation is slow or JS is delayed. The back-to-step-1 screenshot (AP-10-back-to-step1.png) shows the form fully opaque when fields are filled — confirming the issue is animation-state-specific.

2. **Review screen empty (HIGH):** /apply/review shows only the validation banner and guarantee badge. Per CLAUDE.md spec, it should show "every field, every document, cleanly laid out in two-column format." The full summary is completely absent — the page is functionally minimal.

3. **Auth page left panel copy mismatch:** The navy left panel shows "Your documents, handled." as the headline with "Join thousands of NRIs who've stopped fighting government portals." as the subtext. CLAUDE.md specifies the landing headline as "Your OCI card. Your passport renewal. Done in minutes, not weeks." — the auth panel copy is slightly different but on-brand enough. Minor.

4. **Account page missing danger zone:** The account page cuts off before the danger zone section in the viewport. Scrolling reveals it (not tested) — but no DELETE input was found in DOM at all, suggesting it may not be implemented yet.

5. **Auth error element structure:** Wrong-password error renders visually (red alert box with correct message) but uses a DOM structure that doesn't match standard accessibility patterns (`[role="alert"]` or `<p>` with error class). Should use `role="alert"` for screen reader announcements.

6. **Onboarding "0 KB" file size:** The upload zone shows "0 KB · Click to replace" for the test fixture. This is because the test JPEG is a minimal 332-byte file. Not a product bug, but worth noting that the file size display works correctly.

7. **Mobile landing page:** "Sign in" text and logo appear compressed in the navbar at 375px — "avasafe Sign in" runs together visually in the screenshot. The pill nav layout works but is tight.

## UX ISSUES

1. **No AVA voice on /apply/form:** The form correctly shows an AVA bubble on step 1 ("Let's start with your name exactly as it appears on your passport.") but the bubble styling uses the navy left-border design from the spec. However, it uses DM Sans regular, not Playfair Display italic as specified for AVA's voice in CLAUDE.md. AVA's voice should be distinctly italic serif to feel warm and different from UI copy.

2. **Review page missing field summary:** The review page currently only shows validation result (pass/fail) and the guarantee badge. Per CLAUDE.md spec and UX philosophy, the user should be able to review every field before paying. The current UI goes straight from form completion to "Everything looks perfect — Pay $29." Users cannot catch their own mistakes without clicking "Edit application."

3. **No "Why do we need this?" tooltips on form:** The spec mandates a "Why do we need this?" tooltip on every form field. None are present on any step tested (steps 1-3). For a high-stakes application (OCI card), users will have questions about why certain data is needed.

4. **Continue button not clearly disabled on empty form:** On step 1 with empty fields, Continue shows as greyed-out (visually disabled appearance) but there is no visible validation message explaining why. Only after the user manually attempts to proceed does the state become actionable.

5. **Onboarding plan page (/onboarding/plan) "Continue free" routing:** The page loads and shows plan options, but clicking "Continue free" does not navigate to /dashboard within 10 seconds. This may mean the button triggers a Supabase update that hangs, or client-side routing is broken after the click. A user on the free path would be stuck.

6. **Account page shows "—" for full name:** New accounts (or accounts created without a name) show a literal em dash as the full name. This should instead show a placeholder link like "Add your name →" to prompt completion.

## TOP 5 FIXES
Ranked by user impact:

1. **Fix the /apply server crash (BUG-001/002/003)** — The entire application flow (OCI, passport renewal, package, status) is inaccessible due to a shared server-side exception. This is a complete blocker for the core product. Check Supabase queries in the /apply route group for null-handling when `applications` table returns zero rows.

2. **Fix /apply/form invisible form fields on load (BUG-005)** — The form text, labels, and input borders are invisible on initial render due to Framer Motion animation state. Users see a blank page for the first frame. Either set `initial={{ opacity: 1 }}` or ensure the animation completes within 100ms. This is the most important screen in the product.

3. **Implement review page field summary (BUG-012)** — The review screen is specified as showing "every field, every document, cleanly laid out in two-column format." Currently it only shows a validation banner. Users cannot verify their data before paying $29, which will cause trust failures and support requests.

4. **Fix /onboarding/plan "Continue free" routing (BUG-004)** — New users on the free path are stuck after selecting their plan. This blocks every user who doesn't immediately upgrade, preventing them from reaching the dashboard and experiencing the product.

5. **Add FAQ interactivity (BUG-011)** — FAQ items are not keyboard-accessible buttons. For a trust-critical product asking users to upload their passport, FAQ is a primary trust-building element. Items must be `<button>` or `<details>` elements with proper open/close and chevron rotation.
